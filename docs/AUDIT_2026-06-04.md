# Full Audit Report — WorldCup26.world

**Date:** 2026-06-04
**Target:** `worldcup` repository powering **https://worldcup26.world** (Next.js 16 · React 19 · Supabase/Postgres real-money fantasy game)
**Scope:** Entire `main` branch — all `src/` (102 files), 36 SQL migrations, 36 API routes, 8 client components, the new PWA layer, config, CI, release scripts, and docs.
**Type:** Independent security, money-integrity, correctness, compliance, and quality audit.
**Method:** Full source read of the money rails (deposits, withdrawals, settlement, wallet, tickets, agents) and the DB privilege/RLS model; three parallel breadth passes (money/DB, auth/gating/compliance, frontend/PWA/tests/CI); objective verification (`typecheck` / `lint` / `test` / `build` / `npm audit`) on a fresh `npm install`; every High/Medium finding re-verified by hand at `file:line`.

> This report supersedes [`AUDIT_2026-06-03.md`](./AUDIT_2026-06-03.md) for current state. Since that audit the project **closed H-2** (wallet-lock unification + a non-negative-balance trigger), **landed the sender-wallet binding** for manual USDT (former H-1), **added a production CSP**, and shipped new surfaces: a **PWA** (manifest + service worker + install CTA), **user-to-user ticket transfers**, **agent self-registration**, and an **Agent Call** ticket-request flow. This audit re-verifies the closed items and focuses on the new surface. The dominant blockers remain **compliance/product**, not the database.

## 1. Executive summary

The engineering remains **high quality and, in the money layer, notably careful**. Balances are an append-only ledger (`worldcup_wallet_transactions`, every amount `numeric(20,8)`) treated as the single source of truth; deposits, settlement, and admin credit are idempotent; entry creation, wallet transfers, ticket purchases, transfers, and withdrawal debits run inside atomic `SECURITY DEFINER` functions with advisory locks and under-lock balance rechecks. The privileged-RPC lockdown holds: **no money/ticket function is executable by `anon`/`authenticated`/`public`**. Admin authorization is textbook (Google verified-email allowlist + constant-time break-glass secret, fails closed). **Full verification passes: typecheck ✓, lint ✓, 294/294 tests ✓ (67 suites), `npm audit` → 0 vulnerabilities, production build ✓ (36 routes + middleware).**

**What changed for the better since 2026-06-03:** the former High on inconsistent wallet locks (**H-2**) is **closed** — all three debit functions now derive one canonical advisory key and a non-negative-balance trigger backstops the floor; the shared-address deposit gap (**H-1**) now records a required sender-wallet address (PR #17 landed), raising friction; a **production CSP** was added and is enforced by a test.

**What now stands between this code and a real-money launch is still compliance.** There is **no KYC/AML/sanctions screening and no legal classification/licensing in code** — both are explicitly deferred to manual admin process and an honor-system launch checkbox. For a paid-entry, USDT cash-prize contest that is the dominant risk. On the code side, the new launch-relevant findings are: **two ticket-acquisition paths (`tickets/claim`, Agent Call accept) bypass the geo/launch/responsible-play gate stack** because gating is per-route opt-in with no central enforcement (H-1 below), and the **migration history no longer replays cleanly** on a fresh database (M-1 below) — invisible to CI, which runs no migration replay.

## 4. Architecture & money model (recap)

- **Stack:** Next.js 16 App Router (36 API routes, 8 client components), React 19, Supabase Postgres (36 migrations), Vercel. A canonical-host middleware (`src/proxy.ts`) 308-redirects page routes only. Crons: results hourly, point-apply at `:30`, deposit reconcile every 10 min (`vercel.json`). New: a PWA (manifest + `public/sw.js` + install CTA).
- **Identity:** Google-only OAuth. Server routes authenticate the Supabase bearer token and reject non-Google providers (`getAuthProvider`).
- **Balances:** append-only `worldcup_wallet_transactions` ledger; balance = signed sum; no stored balance column to drift.
- **Money in:** (1) per-user KuCoin broker sub-account → unique address → cron `reconcile` credits idempotently on `(provider, external_id)`; (2) shared main-address + user claim (now carrying a recorded sender wallet) → admin verifies against KuCoin → credits. Deposits **auto-convert** full ticket-price chunks into tickets in the DB (`worldcup_credit_deposit`).
- **Money out:** user submits a withdrawal request; admin approves (reserve-then-debit saga, rolls back on RPC failure), then marks paid after a **manual** off-platform USDT transfer.
- **Supply/economy:** entry consumes a ticket; tickets come from admin assignment, wallet purchase, **user-to-user transfer** (new — carries the sender's referral identity), **voucher codes**, or the **Agent Call** flow (new — an active agent mints a price-0 ticket from their finite, admin-assigned code pool after off-platform payment). Settlement turns the awarded leaderboard into an idempotent payout ledger with integer-cents allocation and referral splits.
- **Gating (intended):** every paid action passes geo-eligibility → launch-gate (operator policy + evidence-backed sign-offs) → responsible-play → rate-limit before touching money.

## 5. Critical findings (compliance / launch-blocking)

> Product/compliance blockers, intentionally deferred in code (`docs/COMPLIANCE.md:78-83`). They must be resolved before accepting real money. Unchanged from 2026-06-03.

### 🔴 C-1 — No KYC/AML / sanctions screening in code
Grep for `kyc|aml|sanction|ofac|dob|date_of_birth|passport|identity.?verif|watchlist|pep|screening` across `src/` returns **only two non-enforcing UI strings** (`terms/page.tsx`, `wallet-screen.tsx`). Withdrawal approval (`src/app/api/admin/withdrawals/route.ts`) gates on `requireAdmin` + a mandatory note only; an operator can approve and mark-paid a crypto cash-out with no enforced identity or sanctions check. **Remediation:** integrate a KYC/AML/sanctions vendor at deposit-claim and withdrawal-approval; persist verification state; block payout until cleared.

### 🔴 C-2 — No legal classification or licensing; launch is an honor-system checkbox
`docs/COMPLIANCE.md:11` leaves legal classification (skill vs lottery vs gambling) and licences unchecked. The only enforcement is the `legal_compliance_review` launch sign-off, satisfied by an admin pasting a note + an HTTPS URL (`src/lib/launch-signoff-evidence.ts`). No code verifies a licence exists. **Remediation:** obtain per-market legal classification and licences; encode allowed/blocked jurisdictions into the operator policy (already enforced for paid actions).

## 6. High-severity findings (code)

### 🟠 H-1 — Two ticket-acquisition paths bypass the geo / launch-gate / responsible-play stack (gating is per-route opt-in, no central enforcement)
**Root cause:** the only middleware is `src/proxy.ts`, whose matcher **excludes `/api`** (`src/proxy.ts:20-24`) and only does canonical-host redirects. There is **no global enforcement** of authn/geo/launch-gate/responsible-play on API routes — every paid route re-implements the chain by hand, so a route that forgets it ships ungated. Two already do:

- **`src/app/api/tickets/claim/route.ts:20-66`** — redeeming a voucher code grants a tournament ticket after only `enforceRateLimit` + Google-auth. **Missing** vs. `tickets/purchase`: `enforceGeoEligibility`, `getUserPaidActionGate(...,"ticket")`, `getResponsiblePlayRestriction`, and consent. Consequences: a **self-excluded** user can still acquire a ticket (a responsible-gaming control failure — self-exclusion is meant to block acquisition), and codes become **playable before public launch sign-offs complete**.
- **`src/app/api/agent-ticket-requests/[requestId]/accept/route.ts:19-65`** — an active agent mints a ticket to the **requester**, who never passes geo / launch-gate / their own responsible-play check at issuance. An operator-assisted (higher-volume) version of the same gap.

**Mitigation present:** locking teams in `entries/route.ts` *does* enforce geo + launch-gate + responsible-play + consent, so a self-excluded user cannot ultimately *lock* an entry. The residual harm is acquiring/holding a playable ticket while self-excluded and pre-launch ticket issuance. **Remediation:** add the gate calls to both routes, and introduce a shared `withPaidActionGates(...)` wrapper (or API-level middleware allowlist) so a new paid route cannot ship gate-less.

### 🟠 H-2 — Shared-address USDT deposits still cannot be cryptographically bound to the claiming user
**Where:** `src/app/api/deposits/claims/route.ts`, `…_worldcup_shared_deposit_claims.sql`, `…_manual_usdt_incoming_transfers.sql`.
PR #17 added a required `sender_wallet_address` per claim (a real improvement), but on-chain tx hashes and sender addresses are **public**, so a user can still submit values another user actually sent. The admin "Verify KuCoin" step confirms the deposit *exists at the shared address* but cannot prove the claimant is the on-chain sender (shared address, no memo). The `unique (network, tx_hash)` constraint prevents *double*-claiming, not *mis*-attribution. **Remediation:** prefer the per-user sub-account address model in production; for the shared path, require a verifiable proof of sender control (e.g., a unique sub-cent amount nonce per claim) before credit.

### 🟠 H-3 — Age gate is self-attested, not verified
**Where:** `src/lib/consent.ts`, enforced at `entries/route.ts`. Consent is a single boolean `age_confirmed` (MINIMUM_AGE=18) the user POSTs themselves. Enforced server-side, but cannot stop a minor who clicks "yes." Adequate for a free game; insufficient for a real-money product. **Remediation:** capture DOB and verify age via the same KYC vendor as C-1 before paid actions.

## 7. Medium-severity findings

### 🟡 M-1 — Migration history does not replay cleanly on a fresh database
Two defects make `supabase db reset` / a fresh staging / a clean-room CI migration fail:
1. **Grant-after-drop on a dropped function.** `20260602014000_worldcup_function_execution_lockdown.sql:42` does `drop function if exists public.worldcup_finalize_entry(uuid)`, but the later-sorting `20260602021000_worldcup_function_security_hardening.sql:13-16` does `revoke`/`grant execute on function public.worldcup_finalize_entry(uuid)` (no `IF EXISTS` possible on `GRANT`). Because `…014000…` sorts before `…021000…`, a fresh replay drops the function first, then errors on the grant: *function … does not exist*.
2. **Duplicate migration timestamp.** Two files share `20260602014000` (`…_function_execution_lockdown.sql` and `…_usdt_deposits.sql`), which is an ordering smell on its own.

Production (migrated incrementally, out of strict filename order) is unaffected, but **disaster recovery, fresh environments, and any clean migration test break**. CI runs no migration replay, so this is invisible. **Remediation:** remove the finalize-entry grant from the hardening migration (it's dropped anyway) or guard it in a `DO`/`IF EXISTS` block; de-duplicate the timestamp; add a `supabase db reset` smoke step to CI.

### 🟡 M-2 — PWA service worker caches authenticated / money API responses to disk
**Where:** `public/sw.js:39-49`. The fetch handler is network-first and caches **every** same-origin non-document GET whose `response.ok` (`:42-44`) — which includes client `fetch()` calls to authenticated endpoints (`/api/me/standing`, `/api/withdrawals`, `/api/deposits/address`, `/api/agent/me`, `/api/referrals/me`, …); none set `Cache-Control: no-store/private`. On network failure it serves `caches.match(event.request)` (`:48`), which **matches by URL and ignores the `Authorization` header** — so on a shared device, offline, a navigation to the same URL can serve the **previous** user's cached JSON, and wallet/deposit/withdrawal data sits at rest in Cache Storage. **Remediation:** restrict the SW fetch handler to a static-asset allowlist (it already declares `SHELL_ASSETS`) or skip `url.pathname.startsWith("/api")` and any response carrying `Authorization`/`Set-Cookie`/`Cache-Control: private`.

### 🟡 M-3 — Broken referential integrity on money/identity columns
`from_user_id`/`to_user_id` on `worldcup_wallet_transactions`, `requester_user_id`/`agent_user_id` on `worldcup_agent_ticket_requests` (`20260604100000:13,16`), and `transferred_from_user_id`/`source_referrer_user_id` on `worldcup_tickets` (`20260604154000:9,11`) are free-floating UUIDs with no FK. Ledger/ticket rows can reference nonexistent accounts, weakening auditability for a real-money product. **Remediation:** add FKs (to `auth.users`/`worldcup_referral_profiles`) with appropriate `on delete` behavior.

### 🟡 M-4 — The non-negative balance trigger is not concurrency-safe on its own; its comment overstates the guarantee
**Where:** `20260603120000_worldcup_wallet_lock_unification.sql:44-82`; comment at `:22-23` claims any debit path "cannot drive a wallet below zero, lock or no lock." The trigger reads the balance with a plain `SELECT sum(...)` and **takes no advisory lock**, so two concurrent debits that *don't* go through the canonical lock (a future `admin_debit` insert, or a new RPC that forgets the lock) each evaluate against their own MVCC snapshot and can both pass → negative balance. Today all three debit paths take the canonical lock, so live exposure is low — this is about the durability of the invariant and a comment that invites a future lock-free debit. **Remediation:** take `pg_advisory_xact_lock(worldcup_wallet_lock_key(new.tournament_id, new.from_user_id))` inside the trigger, or soften the comment to state the lock is still required.

### 🟡 M-5 — Raw Postgres `error.message` returned to clients
A user-facing (signed-in, non-admin) route leaks DB driver text on failure: `src/app/api/referrals/me/route.ts:77` returns `tickets.error?.message ?? transactions.error?.message`. Admin routes do the same on write failure: `admin/agents/route.ts:161`, `admin/tickets/route.ts:70`, `admin/prize-pool/route.ts:46`; cron routes at `cron/results/route.ts:32`, `cron/apply/route.ts:20`. Schema/constraint disclosure. (The widespread `error.message?.match(/[A-Z_]{4,}/)` and `error instanceof ValidationError ? error.message` patterns are **safe** — they extract a sentinel code or an intentional validation string.) **Remediation:** map all of these to generic strings.

### 🟡 M-6 — `PAID_ACTION_LAUNCH_TEST_BYPASS` has no production-readiness guard and is not surfaced when ON
When enabled, `isPaidActionLaunchTestAdmin` short-circuits geo + the entire launch-gate to fully-open for any allowlisted admin on every paid route (`src/lib/paid-action-gates.ts:48-70,175-198`). Nothing in `src/lib/production-readiness.ts` flags that the kill-switch is on, and the admin launch-evidence probe computes `adminEvidenceActionsAllowed` through the bypassed gate, so it can show actions "allowed" without real evidence and never warns the bypass is the reason — exactly the failure mode `DEPLOYMENT.md` warns about. Mitigations are good (off by default, requires explicit truthy value + allowlisted email), but nothing automated catches it being left on. **Remediation:** add a readiness check that fails/warns loudly when the bypass is enabled, and surface its state in the probe.

### 🟡 M-7 — Five money/agent routes are still covered only by source-grep tests
`tests/money-route-handlers.test.ts` now *executes* `wallet-transfer` and `tickets/claim` via injectable factories (the right pattern). But `tickets/transfer`, `tickets/purchase`, `agent/me` POST, and `agent-ticket-requests` (+ `/accept`) are plain `POST` exports with no factory, covered only by `readFileSync` + regex suites — a logic regression (e.g., a removed auth/gate check) passes as long as the matched string survives. ~28 of 49 suites are grep-style. **Remediation:** refactor these into injectable factories and add executing handler tests asserting auth/gate ordering — they are real-money paths.

### 🟡 M-8 — Launch-grade Terms/Privacy and CI gaps
Terms & Privacy are present and version-pinned to consent but thin (no governing law, operating entity, dispute/arbitration, prohibited-territories list, AML/KYC terms) — `src/app/terms/page.tsx`, `src/app/privacy/page.tsx`. CI (`.github/workflows/ci.yml`) gates typecheck→lint→test→build but has **no `npm audit`, no migration replay/drift check, no coverage gate** — so a future vulnerable dep or the M-1 replay break won't be caught. **Remediation:** lawyer-drafted legal pages; add `npm audit` + a `supabase db reset` step to CI.

## 9. What is done well (credit)

- **Privileged-RPC lockdown holds.** No money/ticket function is granted to `anon`/`authenticated`/`public` (verified across all 36 migrations); each new RPC carries its own `REVOKE`/`GRANT` and is backstopped by the schema-wide lockdown + `ALTER DEFAULT PRIVILEGES`. Every `.rpc()` call uses the service-role client after authn.
- **H-2 fixed correctly.** One canonical `worldcup_wallet_lock_key` used by every debit path + an `AFTER INSERT` non-negative-balance trigger that cannot be bypassed by the service role (not superuser).
- **Atomic, idempotent money logic.** Advisory locks + `FOR UPDATE SKIP LOCKED` + under-lock balance rechecks; deposits idempotent on `(provider, external_id)`; settlement idempotent on a unique payout key with integer-cents allocation; withdrawal approval is a reserve-then-debit saga that rolls back on RPC failure.
- **Ticket double-spend is structurally prevented** across consume/transfer/purchase by row-level `FOR UPDATE` + `consumed_at is null` rechecks; **agent self-registration cannot escalate** (`active:false`, activation admin-only); ticket supply is bounded by the fixed code pool.
- **Textbook admin auth** — Google verified-email allowlist + constant-time break-glass secret read only from a header, fails closed; launch sign-offs **re-verify payment evidence server-side** rather than trusting a stored flag.
- **Clean client/server boundary** (no secret in any client bundle; `admin-console` imports `production-readiness` as `import type` only), **no XSS sink**, a **production CSP** enforced by a test, security headers applied globally and independently of the proxy, and strong accessibility.
- **Receive-only KuCoin posture** (read-only main key, manual payouts), HMAC-signed requests, constant-time secret compares, fail-closed cron/geo, and a genuine end-to-end `production-smoke.mjs` probe.

## 11. Methodology & limitations

No live database, production environment, or real anon key was available, so the RPC-lockdown and balance-trigger conclusions are asserted from the migrations and passing regression tests rather than a live PostgREST probe — re-run `money-rpc-permissions` against production after `supabase db push`, and run a `supabase db reset` to confirm M-1. The M-1 replay break and M-2 cache exposure are reasoned from source and are high-confidence; H-1's practical impact is bounded by the entry-lock mitigation noted inline.
