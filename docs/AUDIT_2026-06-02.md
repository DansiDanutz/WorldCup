# Full Audit Report — WorldCup 2026

**Date:** 2026-06-02
**Scope:** Entire repository (`worldcup` — Next.js 16 + React 19 + Supabase/Postgres real‑money fantasy game)
**Type:** Independent security, money‑integrity, correctness, and quality audit
**Method:** Full source read of `src/`, all 19 SQL migrations, all 16 API routes, config and CI; objective verification (typecheck / lint / test / build / `npm audit`); two focused deep‑dive passes on the DB layer and the HTTP layer.

> This is a fresh, independent audit. It complements the earlier remediation log in
> [`docs/AUDIT.md`](./AUDIT.md): that document recorded a prior hardening pass; this one
> re‑verifies the current state and surfaces what is **still** open — most importantly a
> database‑layer privilege gap that the well‑built API layer cannot protect against.

---

## 1. Executive summary

The codebase is, on the whole, **high quality and carefully engineered**. Domain logic
(scoring, brackets, prize math, result validation) is pure, well‑typed, and well‑tested.
The HTTP API layer is genuinely strong: every admin route gates on `requireAdmin` before
doing work, the admin secret uses constant‑time comparison and fails closed, money mutations
run inside atomic `SECURITY DEFINER` Postgres functions with advisory locks and balance
checks, and the payout ledger is idempotent. Full verification passes: **typecheck ✓, lint ✓,
72/72 tests ✓, production build ✓.**

There is, however, **one critical structural gap** that undercuts the rest:

> **The money‑moving database functions trust a caller‑supplied user id, are `SECURITY
> DEFINER` in the `public` schema, and are never `REVOKE`d from `anon`/`authenticated`.**
> In a default Supabase project these functions are reachable directly over PostgREST
> (`/rest/v1/rpc/...`) using the public anon key — which ships in the browser bundle. That
> path **bypasses the entire Next.js authorization layer.** An attacker could call
> `worldcup_wallet_transfer` with arbitrary `from`/`to` account ids and move balances
> between any accounts, or call `worldcup_create_entry` to forge entries that skip the
> age/consent and ticket gates enforced only in the API route.

**Today's blast radius is limited** because the wallet is an internal accounting ledger with
no withdrawal rail yet (per [`docs/COMPLIANCE.md`](./COMPLIANCE.md), it "does not move
external funds"). But the moment a payment rail is attached, this becomes catastrophic, so it
must be closed before real money flows. The fix is small and well‑understood (one migration:
`REVOKE EXECUTE ... FROM public` + `GRANT ... TO service_role`).

> ✅ **Remediation shipped (pending apply).** Migration
> [`supabase/migrations/20260602014000_worldcup_function_execution_lockdown.sql`](../supabase/migrations/20260602014000_worldcup_function_execution_lockdown.sql)
> locks `EXECUTE` to `service_role` (closing **C‑1** and **H‑1**) and drops the unguarded
> `worldcup_finalize_entry` (**C‑2**), with a regression test in
> `tests/function-execution-lockdown.test.ts`. It still requires a manual
> `supabase db push` (or Supabase deploy) to take effect on the live database — run the
> §3 verification check afterward to confirm.

## 3. Critical findings

### 🔴 C‑1 — Money RPCs are callable directly via the public anon key (authz lives only in the API layer) — ✅ fix shipped in this PR (pending apply)

**Where:** `supabase/migrations/20260602010500_worldcup_atomic_entry_wallet.sql` —
`worldcup_wallet_transfer` (line 138), `worldcup_create_entry` (line 18); enabler: **no
`GRANT`/`REVOKE` anywhere** in `supabase/migrations/`.

**Evidence:**
```sql
-- 20260602010500_worldcup_atomic_entry_wallet.sql:138
create or replace function public.worldcup_wallet_transfer(
  p_tournament_id uuid,
  p_from_user_id uuid,     -- ← acting account comes from a PARAMETER
  p_to_user_id uuid,
  p_amount numeric, p_note text, p_created_by text
)
returns uuid language plpgsql
security definer            -- ← runs with the owner's privileges
set search_path = public    -- ← exposed by PostgREST as /rest/v1/rpc/worldcup_wallet_transfer
as $$ ... $$;               -- ← body never checks auth.uid()
```

The function never compares `p_from_user_id` to `auth.uid()`; it relies entirely on the
Next.js route (`requireAdmin`) to decide who may call it. But in a **default Supabase
project**:
1. `CREATE FUNCTION` grants `EXECUTE` to `PUBLIC`;
2. `anon` and `authenticated` have `USAGE` on schema `public`;
3. PostgREST exposes every `public` function as an RPC endpoint.

Because these migrations contain **no `REVOKE`**, the production DB almost certainly retains
those defaults — meaning the public anon key (`NEXT_PUBLIC_SUPABASE_ANON_KEY`, shipped to
every browser) can `POST /rest/v1/rpc/worldcup_wallet_transfer` and move balances between
**any** two accounts, bounded only by the sender's balance. The same exposure lets
`worldcup_create_entry` forge entries that skip the **age/consent gate and ticket checks**
that exist only in `src/app/api/entries/route.ts` (the DB function does not re‑check consent).

> **Why the two layers disagree:** the API‑layer review correctly found "no unauthenticated
> money path" — *through Next.js*. The exposure is the **PostgREST side door**, which the API
> code cannot guard. This is the single most important issue in this report.

**Impact:** Unauthorized internal fund movement, forged/locked entries, consent‑gate bypass.
Critical by design; mitigated *today* only by the absence of a withdrawal rail.

**Also exposed by the same root cause (state‑disruption rather than theft):**
`worldcup_settle_payouts`, `worldcup_apply_match_points`,
`worldcup_apply_completed_match_points`, `worldcup_mark_match_result_checked`,
`worldcup_finalize_entry` (see C‑2), `worldcup_rate_limit_hit`.

**Remediation (one migration — test in staging first):**
```sql
-- Lock every privileged routine to the server's service role.
revoke execute on all functions in schema public from public, anon, authenticated;
grant  execute on all functions in schema public to service_role;
-- Prevent future functions from silently re-opening the door:
alter default privileges in schema public revoke execute on functions from public;
```
Trigger functions keep firing (they don't need `EXECUTE`). Optionally also add a defensive
`if auth.uid() is distinct from p_from_user_id then ...` style assertion for any function
intended to act on behalf of a specific user.

**How to confirm in your project (safe, moves no money):** call the transfer RPC with the
same id for `from` and `to` — if execute access is open you'll get the business error
`SAME_ACCOUNT`; if it's already locked down you'll get `permission denied for function`:
```bash
curl -s -X POST "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/rpc/worldcup_wallet_transfer" \
  -H "apikey: $ANON_KEY" -H "Authorization: Bearer $ANON_KEY" \
  -H 'Content-Type: application/json' \
  -d '{"p_tournament_id":"00000000-0000-0000-0000-000000000000",
       "p_from_user_id":"11111111-1111-1111-1111-111111111111",
       "p_to_user_id":"11111111-1111-1111-1111-111111111111",
       "p_amount":1,"p_note":"poc","p_created_by":"poc"}'
# "SAME_ACCOUNT"  -> anon CAN execute (vulnerable)
# permission denied for function -> already locked down (safe)
```

### 🔴 C‑2 — `worldcup_finalize_entry` locks any entry by UUID with no ownership check — ✅ fix shipped in this PR (pending apply)

**Where:** `supabase/migrations/20260601193000_worldcup_game.sql:104‑133`

```sql
create or replace function public.worldcup_finalize_entry(target_entry_id uuid)
returns public.worldcup_entries
security definer set search_path = public as $$
begin
  -- counts picks, then:
  update public.worldcup_entries
  set status = 'locked', locked_at = coalesce(locked_at, now())
  where id = target_entry_id;   -- ← no auth.uid() / ownership predicate
  ...
end; $$;
```

`SECURITY DEFINER`, no `REVOKE`, no ownership guard. Via the same PostgREST exposure as C‑1,
anyone can lock any other player's draft entry (griefing / integrity). The function appears
**superseded** by `worldcup_create_entry` (which locks atomically on creation) and is no
longer called by application code.

**Remediation:** `DROP FUNCTION public.worldcup_finalize_entry(uuid);` if unused, or add an
ownership check and `REVOKE` (covered by the C‑1 migration regardless).

## 4. High‑severity findings

### 🟠 H‑1 — Privilege model relies entirely on Supabase defaults (no `GRANT`/`REVOKE`) — ✅ partially addressed in this PR
> The function‑execution lockdown migration adds explicit `REVOKE`/`GRANT` for routines and an
> `ALTER DEFAULT PRIVILEGES` backstop. Defense‑in‑depth **table‑level** `REVOKE`s on the money
> tables (wallet_transactions, tickets, payouts) remain a recommended follow‑up.
**Where:** all of `supabase/migrations/` (verified: zero grant/revoke statements).
This is the **root enabler** of C‑1 and C‑2. The deny‑by‑default on the
`worldcup_wallet_transactions` / `worldcup_tickets` / `worldcup_payouts` tables works *only*
because RLS is enabled with no permissive policy — there is no explicit `REVOKE` backstop. Make
the privilege model explicit (the C‑1 migration plus `revoke all on table ... from anon,
authenticated` on the money tables for defense‑in‑depth).

### 🟠 H‑2 — Broken referential integrity on money/identity columns
**Where:** `20260602001000_worldcup_google_referrals.sql:4‑5` drops the entries→users FK:
```sql
alter table public.worldcup_entries drop constraint worldcup_entries_user_id_fkey;
```
As a result, user‑id columns across `worldcup_wallet_transactions` (`from_user_id`,
`to_user_id`), `worldcup_tickets`, `worldcup_referrals`, `worldcup_referral_profiles`,
`worldcup_payouts`, and `worldcup_consent` are free‑floating UUIDs with **no FK** to the user
table. Ledger rows can reference deleted/nonexistent accounts, weakening money auditability.
Re‑introduce FKs (to `auth.users`) where the schema allows, or document the deliberate
denormalization and enforce existence in the RPCs.

### 🟠 H‑3 — Admin manual result entry skips the rigorous cross‑field validation used for the automated path — ✅ fixed in this PR
**Where:** `src/app/api/admin/results/route.ts:39‑74` vs `src/lib/result-validation.ts`.
The cron/provider path runs `normalizeResultPayload`, which enforces dozens of consistency
rules (total ≥ 90‑min goals, penalties only for penalty finishes, winner consistent with the
score, etc.). The **admin** route only range‑checks `0–99` and writes the columns directly, so
a trusted‑but‑fat‑fingered admin can persist an internally inconsistent result (e.g.
`finishMethod:"90"` with mismatched totals, or a `winnerTeamId` that lost) — which then flows
straight into scoring and bracket advancement. Reuse `normalizeResultPayload` here so both
paths share one source of truth. (Severity is High for *correctness/payout* impact even though
it requires admin access.)

## 7. What is done well (credit)

This is a deliberately balanced report — the engineering here is above average:

- **Atomic, race‑safe money logic.** `worldcup_create_entry` and `worldcup_wallet_transfer`
  use `pg_advisory_xact_lock` + `FOR UPDATE SKIP LOCKED` + under‑lock balance recheck;
  `worldcup_settle_payouts` is idempotent via a `unique(tournament_id, payout_type, entry_id,
  user_id)` key and "credit only if a new payout row was inserted," with integer‑cents math
  and remainder‑to‑last‑place reconciliation.
- **All money columns are `numeric(12,2)`** — never floating point — with `amount > 0` /
  non‑negative CHECK constraints.
- **Admin auth is textbook.** Constant‑time secret compare (`timingSafeEqual`), header‑only
  secret (never from the body), **fails closed** when the env var is unset, and the primary
  path is a Google verified‑email allowlist.
- **Final‑state RLS is correctly scoped to `auth.uid()`** on entries, entry_teams, referrals,
  referral_profiles, payouts, and consent; the original permissive `using(true)` /
  `with check(true)` policies were properly dropped.
- **Clean server/client boundary** — the service‑role key never reaches a client bundle.
- **Strong validation + typing.** Dependency‑free typed validators reject non‑finite/negative
  amounts; `0` `any`, strict TS, no `console` payload logging; `adminSecret` is `Omit`‑ted from
  the provider result type.
- **Robust operations.** Cross‑instance rate limiter with fail‑soft in‑memory fallback,
  per‑match try/catch in the result cron, 10s `AbortController` timeout on the provider fetch,
  security headers (HSTS, X‑Frame‑Options, etc.), canonical‑host middleware, health check, and
  a CI pipeline (typecheck/lint/test/build) with concurrency cancellation.
- **Genuinely hard domain logic done correctly** — 48‑team knockout progression including
  best‑third‑place allocation via bipartite matching, with thorough result cross‑validation.

## 8. Real‑money compliance posture

Per [`docs/COMPLIANCE.md`](./COMPLIANCE.md), this is a paid prize competition and the
following are **intentionally not built** and remain blockers before accepting real money:
legal classification per jurisdiction, **KYC/AML**, **age gating enforced server‑side**,
**geo‑restriction**, **payment‑rail integration**, responsible‑gaming controls, and tax
handling. Two engineering notes tie into this audit:

- The age/consent gate is enforced in `src/app/api/entries/route.ts`, **not** in
  `worldcup_create_entry` — so the PostgREST side door (C‑1) is also a **consent‑gate bypass**.
  Consent/age should be enforced inside the entry RPC (or the RPC locked down) so the gate
  cannot be skipped.
- The internal `worldcup_wallet_transactions` ledger is **accounting only** and moves no
  external funds today — the practical mitigation that keeps C‑1 from being an immediate cash
  loss. Do not attach a withdrawal rail until C‑1/H‑1 are closed.

## 9. Prioritized remediation roadmap

1. **C‑1 / H‑1 / C‑2** — ship the `REVOKE EXECUTE ... FROM public` + `GRANT ... TO
   service_role` migration; drop or guard `worldcup_finalize_entry`. *(One migration, small,
   high impact.)*

   lockdown) so the gate can't be bypassed.

**Soon (hardening):**
4. **H‑2** — restore FKs / enforce account existence on money‑bearing rows.
5. **M‑1** — scope `worldcup_entry_match_points` reads to the owner.
6. **M‑2** — add a rate limiter to `referrals/me`.
7. **M‑4** — replace raw DB error text with mapped/generic messages (esp. `referrals/me`).
8. **M‑3** — use `timingSafeEqual` for the cron secret.

**Backlog (defense‑in‑depth / polish):**
9. **M‑5 / M‑6 / L‑2 / L‑4** — DB balance invariant, shared locking, idempotency keys,
   ticket partial‑unique constraint.
10. **L‑1 / L‑3 / L‑5 / L‑6** — UUID‑validate `teamIds`, add amount maxima, hide fee config,
    bump `next` to clear the `postcss` advisory.
11. End‑to‑end tests for login → referral acceptance → ticket‑gated entry → result entry →
    bracket advancement → settlement (already noted in `docs/IMPLEMENTATION_PLAN.md`).

## 10. Methodology & limitations

Full static read plus objective build/test/audit runs. Findings cite `file:line`. The C‑1/C‑2
exposure is asserted against **default Supabase grant behavior**; because the migrations
contain no `REVOKE`, this is high‑confidence, but it should be confirmed against the live
project with the safe non‑destructive check in §3 (it moves no money). No live database, no
production environment, and no real anon key were available to this audit, so dynamic
exploitation was not performed. Severity reflects design‑level risk; today's blast radius for
C‑1 is bounded by the absence of a withdrawal rail.
