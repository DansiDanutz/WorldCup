# Full Audit Report — WorldCup26.world

**Date:** 2026-06-03
**Target:** `worldcup` repository powering **https://worldcup26.world** (Next.js 16 · React 19 · Supabase/Postgres real-money fantasy game)
**Scope:** Entire repository — all `src/` (98 files), 29 SQL migrations, 33 API routes, 7 client components, config, CI, build/release scripts, docs, and the live site surface.
**Type:** Independent security, money-integrity, correctness, compliance, and quality audit.
**Method:** Full source read of the money rails (deposits, withdrawals, settlement, wallet, agents, tickets) and DB privilege model; three parallel breadth passes (frontend/boundary, compliance/ops, tests/CI/scripts); objective verification (`typecheck` / `lint` / `test` / `build` / `npm audit`); cross-check against the live site and the in-flight PR #17.

> This report supersedes [`AUDIT_2026-06-02.md`](./AUDIT_2026-06-02.md) for current state. That audit's single **Critical** finding (privileged money RPCs reachable via the PostgREST anon "side door") is now **closed and regression-tested**. Since then the app gained a real money rail — **USDT deposits (KuCoin), a withdrawal workflow, and an agent voucher program** — so this audit re-verifies the closed items and focuses on the new surface. The remaining blockers are now **compliance/product**, not the DB privilege defect.

## 1. Executive summary

The engineering remains **high quality and, in the money layer, notably careful**. Balances are an append-only ledger (`worldcup_wallet_transactions`) treated as the single source of truth; every value is `numeric(20,8)` (never floating point); deposits and payout settlement are idempotent; entry creation, wallet transfers, ticket purchases, and withdrawal debits run inside atomic `SECURITY DEFINER` functions with advisory locks and under-lock balance rechecks. Admin authorization is textbook (Google verified-email allowlist + constant-time break-glass secret, fails closed). **Full verification passes: typecheck ✓, lint ✓, 229/229 tests ✓ (53 suites), `npm audit` → 0 vulnerabilities, production build ✓ (33 routes).**

**The prior Critical is fixed.** Migration `20260602014000_worldcup_function_execution_lockdown.sql` `REVOKE`s `EXECUTE` from `public/anon/authenticated`, grants only `service_role`, adds an `ALTER DEFAULT PRIVILEGES` backstop, and drops the dead `worldcup_finalize_entry`. Every new money RPC additionally carries its **own** explicit `REVOKE`/`GRANT`, and `tests/money-rpc-permissions.test.ts` + `tests/function-execution-lockdown.test.ts` assert it. This is exactly the right remediation.

**What now stands between this code and a real-money launch is compliance, not the database.** There is **no KYC/AML/sanctions screening and no legal classification/licensing in code** — both are explicitly deferred to manual admin process and an honor-system launch checkbox. For a $50 USDT paid-entry, cash-prize contest that is the dominant risk. On the code side, the two findings worth fixing before money flows are a **shared-deposit-address attribution gap** (already being addressed by in-flight PR #17) and an **inconsistent wallet-lock discipline** that leaves the non-negative-balance invariant structurally unguaranteed.

## 4. Architecture & money model

- **Stack:** Next.js 16 App Router (33 API routes, 7 client components), React 19, Supabase Postgres (29 migrations), deployed on Vercel behind a canonical-host middleware (`src/proxy.ts`). Crons: results hourly, point-apply at `:30`, deposit reconcile every 10 min (`vercel.json`).
- **Identity:** Google-only OAuth (PKCE). Server routes authenticate the Supabase bearer token and reject non-Google providers.
- **Balances:** append-only `worldcup_wallet_transactions` ledger (`deposit`, `withdrawal`, `transfer`, `ticket_purchase`, `prize_payout`, `referral_payout`, `admin_credit/debit`); balance = signed sum. No stored balance column to drift.
- **Money in (two paths):**
  1. **Per-user KuCoin broker sub-account** → unique deposit address per user/network → cron `reconcile` matches by address and credits idempotently (`worldcup_credit_deposit`, keyed on `(provider, external_id)`). **Address→user binding is sound.**
  2. **Shared main-wallet address + user-submitted claim** (because TRC-20/ERC-20 USDT carries no memo). User submits `{network, amount, txHash}`; admin verifies against KuCoin and credits. **No cryptographic sender→user binding** — see H-1.
- **Money out:** user submits a withdrawal *request*; admin `approve` records a balance-checked ledger debit (`worldcup_record_withdrawal`), then `mark_paid` after a **manual** off-platform USDT transfer (records the external tx hash). No automated payout rail (KuCoin key is documented read-only).
- **Supply/economy:** entry requires a consumed ticket; tickets come from admin assignment, wallet purchase, or **agent voucher codes** (pre-generated pool, "1 free per 10 paid"). Prize settlement turns the awarded leaderboard into an idempotent payout ledger with integer-cents allocation and referral splits.
- **Gating:** every paid action (`deposit`, `withdrawal`, `ticket`, `entry`) passes geo-eligibility, an evidence-backed launch-gate (`paid-action-gates.ts`), responsible-play, and rate limiting before touching money.

## 5. Critical findings (compliance / launch-blocking)

> These are **product/compliance** blockers, not code defects. The code intentionally defers them (`docs/COMPLIANCE.md`, `docs/PAYMENTS.md`). They must be resolved before accepting real money.

### 🔴 C-1 — No KYC/AML / sanctions screening in code
There is zero identity-verification, source-of-funds, or OFAC/sanctions logic anywhere; the only references are UI copy and prose. Withdrawal approval (`src/app/api/admin/withdrawals/route.ts`) gates on `requireAdmin` + a mandatory admin note only — an operator can approve and mark-paid a crypto cash-out with no enforced identity or sanctions check. For USDT in/out this is a money-laundering and sanctions-exposure hole. **Remediation:** integrate a KYC/AML/sanctions vendor at deposit-claim and withdrawal-approval; persist verification state; block payout until cleared.

### 🔴 C-2 — No legal classification or licensing; launch is an honor-system checkbox
`docs/COMPLIANCE.md` leaves "legal classification (skill vs lottery vs gambling) and licences" unchecked. The only enforcement is the `legal_compliance_review` launch sign-off, satisfied by an admin pasting a note + an HTTPS URL (`src/lib/launch-signoff-evidence.ts`). No code verifies a license exists. A $50-entry, cash-prize, crypto contest is regulated gambling or a paid prize competition in most jurisdictions. **Remediation:** obtain per-market legal classification and any required licences; encode the resulting allowed/blocked jurisdictions into the operator policy (which is already enforced — see §7).

## 6. High-severity findings (code)

### 🟠 H-1 — Shared-address USDT deposits cannot be bound to the claiming user (mis-attribution / theft of an unclaimed deposit)
**Where:** `src/app/api/deposits/claims/route.ts`, `src/app/api/admin/deposit-claims/route.ts`, `…_worldcup_shared_deposit_claims.sql`.
For the shared main-address path, a claim is keyed only on `unique (network, tx_hash)`. **On-chain tx hashes are public**, so any signed-in user can submit a tx hash that another user actually sent (before that user files their own claim). The admin "verify" step matches the tx against KuCoin's deposit list by `txHash + address + amount + currency` (`findMatchingMainDeposit`, `src/lib/kucoin.ts:416`) — it confirms the deposit *exists at the shared address* but **cannot prove the claimant is the on-chain sender**, because the shared address is identical for all users and USDT carries no memo. A "matched" result can therefore mislead an admin into crediting the wrong account. The `unique` constraint prevents *double*-claiming a tx, not *mis*-attributing it.
**Mitigations present:** crediting is manual + requires an admin note; the **per-user sub-account reconcile path is not affected** (it binds address→user). **In flight:** PR #17 ("Require sender wallets for manual USDT reconciliation") adds a required sender-wallet address to each claim — a real improvement, though a determined attacker can read the genuine sender address on-chain too, so it raises friction rather than fully closing the gap.
**Remediation:** prefer the per-user sub-account address model for production; for the shared path, require a verifiable proof of sender control (e.g., a unique sub-cent amount nonce per claim, or signature from the sending address) before credit, and surface the on-chain sender to the admin for cross-check.

### 🟠 H-2 — Inconsistent advisory-lock keys across wallet-debit paths; no DB-level non-negative-balance invariant
**Where:** the four balance-affecting RPCs each take a *different* advisory-lock key:
- `worldcup_wallet_transfer` → `hashtext('{tid}:{from_uid}')` (`…_atomic_entry_wallet.sql:165`)
- `worldcup_purchase_ticket` → `hashtext('{uid}:{tid}:ticket')` (`…_usdt_deposits.sql:145`)
- `worldcup_record_withdrawal` → `hashtext('{tid}:{uid}:withdrawal')` (`…_withdrawal_requests.sql:67`)
- `worldcup_create_entry` → `hashtext('{uid}:{tid}')` (`…_atomic_entry_wallet.sql:45`)

Because the keys differ, two debits of **different types** on the same wallet do not serialize against each other: each reads the same starting balance, each passes its own check, and their combined effect can exceed the balance (overdraw → negative balance). There is also **no `CHECK`/trigger enforcing a non-negative net balance** (confirmed: none in any migration), so nothing structurally prevents it.
**Practical exploitability today is low** — the only *user-triggered, instant* debit is `ticket_purchase`, and concurrent ticket purchases share one key and serialize correctly; the other debits are admin-timed. **But this becomes High the moment a second user-triggered debit exists** (e.g., a self-service/automated withdrawal rail). **Remediation:** use one canonical per-`(tournament,user)` advisory-lock key in *every* balance-mutating function (including settlement), and add a DB-level balance-floor assertion under that lock.

### 🟠 H-3 — Age gate is self-attested, not verified
**Where:** `src/lib/consent.ts`, `src/app/api/consent/route.ts`, enforced at `entries/route.ts:95-107`.
Consent is a single boolean `age_confirmed` + terms version the user POSTs themselves. It is enforced server-side (cannot be skipped), but it cannot stop a minor who clicks "yes." Adequate for a free game; insufficient for a real-money product. **Remediation:** capture DOB and verify age via the same KYC vendor as C-1 before paid actions.

### 🟠 H-4 — Admin emails fully bypass geo + launch-gate; owner email is hardcoded and non-revocable
**Where:** `src/lib/paid-action-gates.ts` (`isPaidActionLaunchTestAdmin`), invoked in every paid route (e.g. `entries/route.ts:79`, `deposits/claims/route.ts:119`, `withdrawals/route.ts:124`); `src/lib/admin-auth.ts:6` `OWNER_ADMIN_EMAIL = "semebitcoin@gmail.com"`.
Any allowlisted admin email skips geo-eligibility and the launch-gate on all paid actions (responsible-play and consent still apply). Intended for launch testing, but it means an admin operates with compliance controls disabled, and the owner email is always an admin regardless of env config — it cannot be revoked without a code change. **Remediation:** scope the test-bypass to a non-production flag/env, and make the owner admin configurable so it can be rotated/revoked.

### 🟠 H-5 — API route handlers are never executed by tests (money routes elevated)
**Where:** `tests/*.test.ts`.
~20 of the 53 suites assert that route/migration **source strings** exist (`readFileSync` + regex), which proves a line is present, not that it behaves correctly (auth ordering, error mapping, edge inputs); a refactor that keeps the string but breaks the logic still goes green. Worse, `src/app/api/admin/wallet-transfer/route.ts` and `src/app/api/tickets/claim/route.ts` have **no test of any kind**, and the gatekeeper libs `paid-action-gates.ts`, `http.ts` (geo/rate-limit), and `scoring.ts` are never executed by a test. For code that moves real balances, this is elevated. **Remediation:** add handler-level tests (invoke the exported `POST`/`GET` with a stubbed service client) for the money routes and execute the gating libs.

## 9. What is done well (credit)

- **The prior Critical was fixed correctly** — lockdown migration + per-function `REVOKE`/`GRANT` + two regression tests; new money RPCs follow the same pattern by default.
- **Atomic, idempotent money logic.** Advisory locks + `FOR UPDATE SKIP LOCKED` + under-lock balance rechecks; deposits idempotent on `(provider, external_id)`; settlement idempotent via a unique payout key with integer-cents allocation and remainder-to-last-place.
- **Ledger as single source of truth**, all amounts `numeric(20,8)` with `amount > 0` checks — no stored balance to drift, no floating point.
- **Textbook admin auth** — Google verified-email allowlist + constant-time break-glass secret read only from a header, fails closed.
- **Server-enforced, evidence-backed launch gating.** Payment sign-offs validate against real deposit/withdrawal rows with KuCoin verification and are non-waivable (`launch-signoff-evidence.ts`); geo + launch-gate + responsible-play + rate-limit guard *every* paid action.
- **Clean client/server boundary** (no secret in any client bundle), **no XSS surface**, and **strong accessibility** (ARIA carousel, keyboard nav, reduced-motion, alt text).
- **Receive-only KuCoin posture** (read-only main key, manual payouts), constant-time secret compares, fail-closed cron/geo, security headers, health check, and a genuine end-to-end `production-smoke.mjs` probe.
- **Hard domain logic done right** — 48-team knockout progression with best-third-place bipartite matching, and result cross-validation now shared by both the cron and admin paths.

## 10. Real-money launch readiness

**Safe to launch in free/pre-launch mode (as the live site is today):** yes — paid actions are gated off, and the architecture is sound. The site correctly shows "Prize Pool: TBA / Paid actions open after launch approvals."

**Before flipping on real money, in order:**
1. **C-1 / H-3** — KYC/AML + sanctions screening and verified age at deposit-claim and withdrawal-approval.
2. **C-2** — legal classification + licences per market; encode allowed/blocked jurisdictions into operator policy.
3. **H-1** — adopt the per-user sub-account deposit model in production (or land PR #17 *and* add proof-of-sender) so deposits can't be mis-attributed.
4. **H-2** — unify wallet-lock keys + add a DB-level non-negative-balance assertion before any self-service withdrawal exists.
5. **H-4** — scope the admin gate-bypass to a non-prod flag; make the owner admin revocable.
6. **M-2** — lawyer-drafted Terms/Privacy (jurisdiction, entity, dispute, AML, RG policy).
7. **H-5 / M-6** — executing tests for the money routes + `npm audit` in CI.

**Backlog:** M-1 (FKs), M-3 (CSP + middleware gating), M-5 (error-text mapping in admin routes), M-7 (voucher entropy), M-8 (idempotency keys), and the Low items.

## 11. Methodology & limitations

Full static read plus objective `typecheck`/`lint`/`test`/`build`/`npm audit` runs on the current branch; findings cite `file:line`. The live site was inspected via fetch (pre-launch state confirmed). No live database, production environment, or real anon key was available, so the C-1 closure is asserted from the migrations and the passing `money-rpc-permissions` test rather than a live PostgREST probe — re-run that probe against production after `supabase db push` to confirm the lockdown is applied there. The H-1 attribution gap and H-2 lock-key gap are reasoned from the source and are high-confidence; H-2's *practical* exploitability today is bounded by there being only one user-triggered debit path. PR #17 was reviewed from its description, not its diff (it targets a different branch).
