# WorldCup26 UX Audit — State-Based App Strategy

Date: 2026-06-06  
Scope: Public landing flow, logged-in unpaid activation, paid/picked dashboard, referral and agent income surfaces.

## Executive Summary

WorldCup26 already has the core pieces: a strong visual identity, a simple pick-3 game loop, referral economics, agent ticket selling, Google auth, manual ticket assignment, USDT deposit support, and leaderboard/standing views.

The main UX problem is not missing functionality. The main problem is that the app currently behaves like one long dashboard for every user. The product should instead behave like three different apps depending on user state:

1. Logged out visitor: convince them to sign up.
2. Logged in but unpaid / not activated: make them get a ticket through Agent Call, cash, or USDT deposit.
3. Paid and locked: make them return, track rank, refer friends, and become or use agent tools for passive income.

Until the interface fully changes its priority by state, users will see too many cards that are technically correct but emotionally wrong for the moment they are in.

## State 1: Logged Out Visitor

### User Intent

The visitor has not trusted the product yet. They need to understand:

- What is the game?
- How do I win?
- Why should I join before the World Cup starts?
- Why should I use a referral link?
- Why should I care about agent tickets?

### Current Strengths

- The hero swiper already explains scoring, prize pool, examples, coefficients, invites, and agent deal.
- Referral landing supports inviter codes and Google sign-in.
- Prize pool and player/leaderboard concepts are visible before login.
- The visuals are becoming distinctive and app-like.

### Current Problems

- Visitor lands into a full app dashboard with pick, entry, invite, leaderboard, rules, and schedule concepts competing at once.
- The primary action is not always obvious. Sometimes "Pick 3 teams" feels primary, sometimes "Login / Register", sometimes "Invite".
- Logged-out invite and leaderboard sections expose states that only become useful after signup.
- Agent deal is presented as a slide, but not as a clear reason to sign up now.

### Recommendation

Make the logged-out route a conversion experience:

- First viewport: hero slides plus one fixed CTA group: "Join with Google" and "I have a referral code".
- Secondary proof: prize pool, top payout preview, 48 teams, 104 matches.
- Referral value: "Join through an inviter, then your own invites can earn 5%."
- Agent value: "Want to sell tickets? 10 paid tickets = 1 free code."
- Only show leaderboard as a preview teaser, not as a work dashboard.
- Hide or collapse wallet, entry internals, account stats, and operational cards until login.

## State 2: Logged In, Not Yet Paid / Not Activated

### User Intent

The user has completed Google signup. Now the only important question is:

"How do I get the ticket that lets me lock my 3 teams?"

### Current Strengths

- The app knows whether the user is signed in.
- The app knows ticket availability, wallet balance, entry status, and referral code.
- Agent Call exists with code/email and request-ticket behavior.
- USDT deposit flow exists with sender wallet storage and admin review.

### Current Problems

- Payment activation is not the first clear experience after signup.
- Agent Call is buried inside the Entry ticket requirement card.
- USDT deposit lives more naturally in Wallet, but users who just signed up may not know to go there.
- "Wallet balance", "Ticket price", and "Set by admin" feel like admin/backend terms, not user guidance.
- Picking teams before ticket activation can be fun, but it can distract from the real next step: get one assigned ticket.

### Recommendation

Create an activation dashboard for signed-in unpaid users:

1. Header: "Account ready. Get 1 ticket to lock your teams."
2. Progress: Google account connected -> Pay/receive ticket -> Pick 3 teams -> Lock entry.
3. Primary card: "Get your ticket" with two clear tabs or choices:
   - Agent Call: enter agent code/email, pay the agent directly, agent accepts after payment.
   - USDT deposit: add sender wallet first, then show ERC20/TRC20 address and QR, submit proof, wait for admin credit.
4. Keep team picking visible but secondary: "Preview your 3 teams while ticket is pending."
5. Move referral link below activation: useful, but not the main job yet.

Suggested user copy:

- "Use an agent code if someone sold you a ticket."
- "Use USDT if you want admin to verify payment and assign your ticket."
- "Your deposit address appears only after you save the wallet you will send from."

## State 3: Paid User With Locked Picks

### User Intent

This user already paid and picked. They need reasons to come back and reasons to invite others.

The emotional product changes from "join the game" to "grow my position and passive income."

### Current Strengths

- My Standing shows rank, points, picks, payout projection, and referral list.
- Agent card can reveal the next code to sell.
- Referral card can show referred players and their leaderboard positions.
- Leaderboard and payout preview exist.

### Current Problems

- The passive income story is not the primary post-pick experience.
- Referral and agent tools appear as separate panels instead of one clear "grow your upside" loop.
- Agent codes, referrals, and leaderboard positions are good, but they need a stronger hierarchy after the entry is locked.
- Wallet/deposit/payout content should become status/settings after locking, not the main emotional screen.

### Recommendation

Create a locked-entry dashboard:

1. Top card: "Your WorldCup position" with rank, points, projected payout, and 3 picks.
2. Second card: "Grow passive income" with:
   - Copy referral link.
   - Send WhatsApp invite.
   - Referral list with each player rank.
   - Agent status and next code to sell if agent is active.
3. Third card: "Match progress" with next relevant matches for the user's teams.
4. Wallet/payout card becomes compact status:
   - Ticket paid.
   - USDT sender wallet saved or not saved.
   - Final payout after tournament ends.

Suggested user copy:

- "Your picks are locked. Now grow your upside."
- "Every friend who joins through you can add passive income if they win."
- "Agent mode: sell ticket codes and earn free commission codes."

## Priority Findings

### P0 — State Mixing

The app should not show the same section priority for visitors, unpaid users, and paid/locked users.

Fix: introduce a single journey mode in `Dashboard`, for example:

- `visitor`: no Google session.
- `activation`: signed in, no assigned ticket.
- `picking`: signed in, ticket available, no locked entry.
- `locked`: signed in, entry locked.
- `agent`: overlay capability, not a separate primary state.

Then render or reorder cards by mode.

### P0 — Activation CTA Is Not Strong Enough

After signup, the app must push ticket activation harder than any other feature.

Fix: make "Get your ticket" the top signed-in unpaid card. Agent Call and USDT deposit should sit there, not only in Wallet or deep inside Entry.

### P0 — Post-Lock Passive Income Is Not Prominent Enough

After a user pays and locks picks, the best retention loop is referrals and agent selling.

Fix: make "Grow passive income" the second card after My Standing. It should combine referral link, referred players, agent status, and next sellable code.

### P1 — Navigation Should Change by State

Visitor navigation should focus on slides, prize, invite, agent deal, login.  
Unpaid navigation should focus on Get Ticket, Pick Teams, Wallet, Help.  
Locked navigation should focus on My Standing, Leaderboard, Referrals, Agent, Matches.

### P1 — User Copy Still Sounds Operational

Examples to replace:

- "Wallet balance / Internal funds" -> "Available credit".
- "Ticket price / Set by admin" -> "Entry ticket price".
- "Tickets available / assigned" -> "Entry tickets ready".

### P1 — Agent Deal Needs a Clear Signup Path

The visitor sees the agent deal, but the next step is not obvious enough.

Fix: add "Become an agent" CTA that takes signed-out users to signup, then signed-in users to Agent Wallet registration or agent ticket deposit.

### P2 — Empty Leaderboard Should Sell the Opportunity

Before points or locked entries exist, the leaderboard should feel like "be first", not an empty table.

Fix: in empty state, show:

- "No locked entries yet."
- "First players get visible rank as soon as matches begin."
- CTA: "Join before June 18."

## Recommended Screen Order by State

### Visitor

1. Hero swiper / story slides.
2. Join card with referral path.
3. Prize/payout preview.
4. Referral and agent deal explanation.
5. Rules summary.
6. Leaderboard teaser.
7. Schedule.

### Signed In, No Ticket

1. Account ready banner.
2. Get Ticket card: Agent Call + USDT deposit.
3. Pick preview.
4. Referral link, secondary.
5. Rules.
6. Schedule.

### Signed In, Ticket Ready, Not Locked

1. Pick 3 teams.
2. Entry lock card.
3. Ticket status.
4. Referral link.
5. Leaderboard preview.

### Locked Entry

1. My Standing.
2. Grow passive income: referrals + agent.
3. Leaderboard.
4. My team match progress.
5. Wallet/payout status.
6. Rules and schedule.

## Implementation Plan

### Phase 1 — Journey Mode Router

- Add `userJourneyMode` in `Dashboard`.
- Use existing data: `signedInWithGoogle`, `myAccountStatus`, `ticketsAvailable`, `accountEntry`, `accountEntry.locked`.
- Reorder existing sections by mode without changing backend behavior.

### Phase 2 — Activation Card

- Extract Agent Call and USDT sender-wallet/deposit entry points into one top-level "Get your ticket" card for signed-in unpaid users.
- Keep the current API calls.
- Keep admin manual verification language simple and clear.

### Phase 3 — Locked Dashboard

- Make `MyStanding` the top screen for locked users.
- Combine referral and agent prompts into a strong income card.
- Keep Wallet as a compact status surface.

### Phase 4 — Copy Pass

- Replace admin/internal labels with player-facing labels.
- Keep rules and payment language simple.
- Avoid policy/blocker language on user surfaces unless it is actually blocking an action.

### Phase 5 — Visual QA

Verify all four important states:

- Logged out.
- Logged in, no ticket.
- Logged in, ticket available, no lock.
- Logged in, locked entry.

Each state needs mobile screenshots and browser checks. A logged-out browser check is not enough for signed-in states.

## Final Direction

The best UX is not more cards. The best UX is fewer, sharper cards per user state.

WorldCup26 should feel like:

- Visitor: "This is exciting. I should join."
- Signed-in unpaid user: "I know exactly how to get my ticket."
- Paid locked user: "I am in the game. Now I can earn more by inviting and selling codes."
